# S3 Backup

> Route `/s3` · source `src/pages/control/S3BackupPro.tsx`

![S3 Backup](../screenshots/s3.png)

A configuration **helper** for bunqueue's S3-compatible backup feature. It does not turn backups on — bunqueue reads its backup settings from **server-side environment variables** — this page just gives you a form to assemble those values, stored locally in your browser.

::: warning READ THIS FIRST
Nothing on this page changes the running server. bunqueue OSS enables S3 backups via the server process's environment (`S3_BACKUP_ENABLED`, `S3_BUCKET`, `S3_REGION`, …). This form is a local convenience for composing that config; the values are saved to `localStorage`, never pushed to bunqueue. See [Gotchas](#gotchas).
:::

## What it shows

The page is a `PageHeader` ("S3 Backup" · "Automatic backups to S3-compatible storage.") followed by a status banner, an explanatory notice, a **Connection Settings** card, and a **Backup History** card.

**Status banner** (top strip with the S3 icon):

| Element | Meaning |
| --- | --- |
| Title | `Backup configured` when a bucket name is set, otherwise `No backup configured`. |
| Subtitle | `Target: <bucket>` when configured, otherwise `Set up S3 to protect your queue data`. |
| Badge | Green `Ready` pill when configured; amber `Configure` pill when not. |

"Configured" is derived purely from whether the **Bucket name** field is non-empty (`s3.bucket.trim() !== ''`). It is not a check against the server or against S3 — entering any bucket name flips the banner to green.

**Notice line** (below the banner): a static reminder that bunqueue reads S3 settings from server environment variables (`S3_BUCKET`, `S3_REGION`, …), that this form only helps assemble that configuration, and that it is stored locally in your browser.

**Connection Settings card** — a form (subtitle: "Works with AWS S3, Cloudflare R2, MinIO, DigitalOcean Spaces, and any S3-compatible provider.") with these inputs:

| Field | Type | Default / placeholder | Notes |
| --- | --- | --- | --- |
| Endpoint | text | `https://s3.amazonaws.com` (placeholder) | starts empty |
| Region | text | `us-east-1` (actual default value) | |
| Bucket name | text | `my-bunqueue-backups` (placeholder) | full-width; drives the "configured" state |
| Access key ID | text | `AKIA…` (placeholder) | in-memory only (see below) |
| Secret access key | password | masked | in-memory only (see below) |
| Backup schedule | select | `Disabled` | options: Disabled / Every 6 hours / Every 12 hours / Every 24 hours (`disabled` \| `6h` \| `12h` \| `24h`) |
| Path prefix (optional) | text | `backups/production/` (placeholder) | starts empty |

A caption under the credential fields states: *"Keys are kept in memory only and cleared on reload."*

**Backup History card** — always renders the `EmptyState` **"No backups yet"** with the hint "Configure your S3 connection and run a backup to get started." There is no list, table, or history data behind it; the dashboard cannot trigger or read backups, so this empty state is permanent.

## What you can do

| Action | Effect | Confirm? |
| --- | --- | --- |
| Edit any field | Writes straight into the `s3Store` (Zustand) via `s3.set({ … })` on every keystroke. | No |
| **Save Configuration** (accent) | Shows a transient `Saved locally (keys excluded)` message for ~2s. Persistence itself happens continuously via the store's `persist` middleware — this button only surfaces the confirmation and does **not** call the server. | No |
| **Test Connection** | Calls `bq.storage()` and reports inline: `Server storage reachable` (success) or `Server disk is full` (`diskFull` true); on a thrown error it shows the error message in danger red. | No |
| **Backup Now** | **Permanently disabled.** Tooltip: "Trigger a manual backup — requires server-side S3 config". There is no bunqueue endpoint to trigger a backup. | n/a (disabled) |

::: info What "Test Connection" actually tests
It checks the **server's own disk** via `GET /storage` — not your S3 endpoint, bucket, or credentials. A green "Server storage reachable" means bunqueue's process disk is fine, nothing about whether the S3 values in the form are valid. There is no S3 credential validation anywhere on this page.
:::

No inline validation runs on any field — there are no required-field checks, no format checks on the endpoint URL or keys, and no error surface tied to the form itself. The only "state" the form produces is the green/amber banner (bucket-name presence) and the two transient messages (Saved / Test result).

## States & gating

- **Loading:** none. The page renders synchronously from `localStorage`-backed store state; there is no fetch on mount, so no spinner or skeleton.
- **Empty:** the **Backup History** card is always the "No backups yet" empty state — there is no data source, so it never populates.
- **Not configured:** amber `Configure` badge and the "No backup configured" copy until a bucket name is entered. This is cosmetic only; all form fields and the Save/Test buttons remain usable regardless.
- **Test result / offline:** `Test Connection` is the only server-touching control. If bunqueue is unreachable (or returns `{ ok:false }` in a way `bq.call` throws on), the thrown error's `message` is shown in red next to the buttons. If reachable, it shows reachable/disk-full based on `data.diskFull`.
- **Backup Now:** always disabled — never enabled by any state on this page.

This page has **no** job-state action gating (`src/lib/jobActions.ts` does not apply here); it has no jobs, queues, or per-item actions.

## Behind the scenes

- **API calls:** exactly one — `bq.storage()` → `GET /storage`, fired only when you click **Test Connection**. Client used: **`bq`** (`src/lib/bq.ts`). No polling, no SSE, no fetch on mount.
- **Response shape:** `/storage` is **wrapped** — `{ ok, data: { diskFull, error, since } }` (`StorageStatusFlat`). The code reads `r.data?.diskFull`. `ok` here is `true` on success and `bq`'s `storage()` correctly types the `data` wrapper. Note `/storage` never returns a `path` field (a common wrong assumption from the classic pages).
- **State store:** `useS3Store` (`src/components/dashboard/stores/s3Store.ts`), a Zustand store with `persist` under the `localStorage` key **`bq-dash-s3`**. A `partialize` (`persistedS3State`) writes **only** `endpoint`, `region`, `bucket`, `schedule`, and `pathPrefix`. The credentials (`accessKeyId`, `secretAccessKey`) are **deliberately excluded** from persistence — they live in memory for the session and are gone on reload, so an AWS secret is never sitting in plaintext at rest.

## Gotchas

::: warning Design limitation, not a bug
Per `docs/known-issues.md`, **S3 backup cannot be configured from the dashboard.** bunqueue reads `S3_BACKUP_ENABLED`, `S3_BUCKET`, etc. from the server process's environment. This form is a local-only helper for assembling that env config to set elsewhere — it has no effect on the running server, and **Backup Now** is permanently disabled.
:::

- **"Save Configuration" saves to your browser, not the server.** It persists the non-secret fields to `localStorage` (`bq-dash-s3`) and shows a 2-second confirmation. It does not validate anything and does not reach bunqueue.
- **"Test Connection" tests the server's disk, not S3.** Success ("Server storage reachable") says nothing about whether the endpoint/bucket/keys you entered are correct — it only reflects `/storage`'s `diskFull` flag.
- **The green "Ready" badge is not a health check.** It flips solely on the bucket name being non-empty; it does not mean backups are running or that credentials work.
- **Backup History is permanently empty** — the dashboard has no endpoint to list or trigger backups, so the "No backups yet" state never changes.
- **Credentials are session-only by design.** Access key ID and secret access key are never written to `localStorage`; reloading the page clears them (they are excluded via `partialize` — see [known-issues.md](../known-issues.md), "S3 secret no longer persisted"). Re-enter them each session.
- A read-only classic variant lives at `/s3-classic` (`src/pages/S3Backup.tsx`); it reads `/storage` through the older `api.ts` client whose shape is wrong (masks a disk-full condition). This Pro page uses the correct `bq.storage()` — prefer `/s3`.
